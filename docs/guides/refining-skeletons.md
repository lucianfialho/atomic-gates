# Refining converter skeletons into production state machines

This guide walks you through upgrading a `skill.yaml` that was
auto-generated by `lib/import_skill.py` from a `SKILL.md` into a
production-ready state machine with schema-validated transitions,
structured output, and branching logic where it matters.

If you haven't authored a state machine before, read
[Authoring state-machine skills](./authoring-state-machines.md) first —
this guide assumes you understand the layout and the runtime.

---

## What a skeleton is — and is not

When you run the converter:

```bash
python3 lib/import_skill.py skills/my-skill/SKILL.md \
  -o skills/my-skill/skill.yaml
```

It produces a **mechanical first draft**:

- Every `## ` heading in the markdown becomes a state, in document order
- Each state's `agent_prompt` is the verbatim section body, prefixed
  with `# TODO: refine this prompt`
- Meta-headings like `## Red Flags` or `## Rules` are flagged with
  `# WARNING: heading looks like documentation, not a fase`
- Transitions are a linear chain terminating in `done`
- No `output_schema`, no `when` conditions, no `gate` scripts

The skeleton **validates against** `schemas/skill-machine.schema.json`
and **runs** on the atomic-gates runner. It produces audit-trail runs
and executes the full prose body turn-by-turn. That's a real,
functional state machine — just a rigid one.

What the skeleton **does not** do:

- Decide which sections are fases vs. documentation
- Infer output shapes each state should produce
- Understand loops (TDD's RED → GREEN → REFACTOR → RED)
- Understand branches (different next state based on output content)
- Distinguish required from optional fields in the agent's output

Refinement is the process of closing those gaps — one state at a time,
guided by real usage.

---

## When to refine

Don't refine a skeleton you haven't used. You won't know where the
friction is until the agent has tried to execute it a few times and
either worked around something or produced output you'd reject.

**Signs a skeleton is ready for refinement:**

- The skill is invoked regularly (weekly or more), and its output is
  something downstream code or humans depend on
- You've seen the agent produce valid markdown output that doesn't
  match your mental model of what the state should have emitted
- Two different states need to read each other's outputs and you're
  tired of referring to them by state name
- The skill has natural branches (success/fail, complete/incomplete)
  that the linear chain can't express

**Signs to leave the skeleton alone:**

- The skill is used once a month and works fine as-is
- You're considering refinement because "it feels cleaner," not because
  of real friction
- The source SKILL.md is still in flux — refine after the prose
  stabilizes, not before

Refinement is a cost. Pay it when it buys something.

## The six phases of refinement

Work through the phases in order. Each phase is independent — you can
stop after any of them and commit a partial refinement.

### Phase 1 — Prune meta-headings

The converter flags documentation-shaped headings with `# WARNING`
comments. Your first pass is to delete the states that correspond to
meta content and re-link transitions around them.

**Example:** a skeleton with these states:

```yaml
states:
  overview:          # WARNING: meta
  red_flags:         # WARNING: meta
  phase_1_fetch:
  phase_2_implement:
  rationalization:   # WARNING: meta
  done:
    terminal: true
```

After pruning:

```yaml
states:
  fetch:
    transitions:
      - to: implement
  implement:
    transitions:
      - to: done
  done:
    terminal: true
```

The content from `overview` and `red_flags` can be moved into the
`fetch` state's `agent_prompt` as context, or dropped entirely if the
agent doesn't need it at runtime. Documentation belongs in `SKILL.md`
for human readers, not in state prompts.

### Phase 2 — Rename states to match their purpose

Converter state IDs are `slugified_headings`. They're often clunky
(`phase_1_fetch_the_thing`, `step_2_implement_properly`). Rename to
short, descriptive identifiers:

```yaml
# Before
phase_1_fetch_the_thing:
  transitions:
    - to: step_2_implement_properly

# After
fetch:
  transitions:
    - to: implement
```

Update `initial_state` and every `transitions.to` reference. Keep the
human-readable context in `description`:

```yaml
fetch:
  description: Fetch issue body and PR diff from GitHub
  transitions:
    - to: implement
```

### Phase 3 — Refine agent_prompts

The raw prompt is the section body copied verbatim. It includes
formatting that was meant for humans reading markdown — bullet lists,
example code blocks, section headers. The agent parses fine but the
prompt is longer than it needs to be.

Rewrite each prompt to be **imperative, concrete, and terminal**:

- State the task in one short sentence
- Reference prior outputs via `{{output.<field>}}` or
  `{{outputs.<state>.<field>}}`
- Declare exactly what the output YAML should look like
- End with the output path: "Emit YAML at `{{output_path}}` with shape: ..."

**Before (converter output):**

```yaml
agent_prompt: |
  # TODO: refine this prompt. The converter copied the section body verbatim.

  Here you need to understand what the issue is asking for. Read the
  issue body carefully. Extract every requirement that the issue
  mentions, whether it's in the body directly or in a comment.

  Requirements might look like acceptance criteria, bulleted lists,
  or just prose. Use your judgment.

  ## Tip

  Don't include out-of-scope requirements.
```

**After:**

```yaml
agent_prompt: |
  Read {{output.fetch.issue}}. Extract every discrete requirement
  from the body and comments. Ignore out-of-scope mentions.

  Emit YAML at {{output_path}} with this shape:
    requirements:
      - id: req-N
        statement: string
        source: "issue body" | "comment by X"
```

Tighter, directive, output-focused.

### Phase 4 — Add output_schema to structured states

For every state that produces data the next state will read, declare
an `output_schema`. This is where the runtime guarantees kick in — the
runner rejects malformed output and re-injects the state's prompt.

Create a JSON Schema file at `skills/<name>/schemas/<state>.output.schema.json`:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "additionalProperties": false,
  "required": ["requirements"],
  "properties": {
    "requirements": {
      "type": "array",
      "items": {
        "type": "object",
        "additionalProperties": false,
        "required": ["id", "statement", "source"],
        "properties": {
          "id": { "type": "string", "minLength": 1 },
          "statement": { "type": "string", "minLength": 1 },
          "source": { "type": "string", "minLength": 1 }
        }
      }
    }
  }
}
```

And reference it from the state:

```yaml
extract_requirements:
  agent_prompt: |
    Read {{output.fetch.issue}}. Extract every discrete requirement...
  output_schema: skills/my-skill/schemas/extract_requirements.output.schema.json
  transitions:
    - to: check_coverage
```

Two rules of thumb for the schema:

1. **Always set `additionalProperties: false`** — prevents the agent
   from sneaking extra fields that downstream states don't expect
2. **Mark a field required if any downstream state references it** —
   otherwise the machine can pass an invalid-looking output forward

See [`schemas/skill-machine.schema.json`](../../schemas/skill-machine.schema.json)
for the subset supported by the validator (`type`, `required`,
`properties`, `additionalProperties`, `items`, `enum`, `minLength`,
`minItems`, `minProperties`, `uniqueItems`, `const`). No `$ref`,
no `allOf`/`anyOf`/`oneOf`.

### Phase 5 — Add `when` conditions on transitions

A linear chain works for state machines where every state runs every
time. The moment a state has an actual decision — "if COMPLETE, go
here; if INCOMPLETE, go back to implement; if NEEDS_DISCUSSION, escalate" —
you need conditional transitions.

Each transition can have a `when` expression evaluated against the
current state's output. The first transition whose `when` matches
wins:

```yaml
emit_verdict:
  agent_prompt: |
    Aggregate findings into a final verdict.
    Emit at {{output_path}}: { verdict: COMPLETE | INCOMPLETE | NEEDS_DISCUSSION }
  output_schema: skills/my-skill/schemas/emit_verdict.output.schema.json
  transitions:
    - to: done
      when: "output.verdict == 'COMPLETE'"
    - to: implement
      when: "output.verdict == 'INCOMPLETE'"
    - to: needs_human
      when: "output.verdict == 'NEEDS_DISCUSSION'"
```

Supported operators: `==`, `!=`, `<`, `<=`, `>`, `>=`. Literals can be
quoted strings, numbers, `true`, or `false`. No compound logic
(`and`/`or`) — if you need it, split into multiple transitions in the
right order (first match wins).

**The runner validates the output before evaluating transitions.** A
state with an invalid output never reaches its `when` expressions —
the machine retries the state with the validation error as context.

### Phase 6 — Smoke test and iterate

After every phase (or every few), run the machine standalone to catch
regressions:

```bash
mkdir -p /tmp/refine-smoke
CLAUDE_PLUGIN_ROOT=$HOME/Code/atomic-gates \
CLAUDE_PROJECT_DIR=/tmp/refine-smoke \
  python3 lib/runner.py <<<'{"tool_name":"Skill","tool_input":{"skill":"atomic-gates:my-skill","args":"..."}}'
```

The runner prints the first state's `additionalContext`. Manually
write valid output to the indicated path, then advance:

```bash
CLAUDE_PLUGIN_ROOT=$HOME/Code/atomic-gates \
CLAUDE_PROJECT_DIR=/tmp/refine-smoke \
  python3 lib/runner.py <<<'{"tool_name":"Skill","tool_input":{"skill":"atomic-gates:my-skill","args":"run_id=<id>"}}'
```

Walk the whole machine. Every state. Try an invalid output at least
once per schema-validated state to confirm the rejection message is
clear.

When the smoke test passes cleanly end-to-end, run the skill in a live
Claude Code session and watch what the agent does with it.

## A worked example: refining `validate-issue`

Suppose you started with the converter skeleton for `validate-issue`
and want a production version. The skeleton has eight states (one per
`## ` heading in the markdown): `overview`, `find_the_linked_issue`,
`read_the_issue_requirements`, `read_the_pr_diff`,
`compare_requirements_to_implementation`, `output_format`, `rules`, `done`.

### After Phase 1 — prune

```yaml
states:
  fetch_linked_issue:
  extract_requirements:
  fetch_pr_diff:
  check_coverage:
  emit_verdict:
  done:
    terminal: true
```

`overview`, `output_format`, and `rules` are meta — deleted. Their
content (especially the verdict enum and the rules about what counts
as a "met" requirement) gets moved into `emit_verdict`'s agent_prompt
as imperative instructions rather than documentation.

### After Phase 2 — rename

Already done in Phase 1 (the names above are post-rename). Originals
were things like `find_the_linked_issue` → `fetch_linked_issue`, and
long variations collapsed to short identifiers.

### After Phase 3 — refined agent_prompts

Each prompt is rewritten to be imperative and reference prior outputs
by structured path.

### After Phase 4 — output_schema per state

Four schema files:

```
skills/validate-issue/schemas/
├── fetch_linked_issue.output.schema.json
├── extract_requirements.output.schema.json
├── check_coverage.output.schema.json
└── emit_verdict.output.schema.json
```

`fetch_linked_issue` output has `{ issue: { title, body, number }, comments: [...] }`.
`extract_requirements` output has `{ requirements: [{ id, statement, source }] }`.
`check_coverage` output has `{ coverage: [{ requirement_id, status, evidence }] }`.
`emit_verdict` output has `{ verdict: enum, met: [ids], missing: [ids], summary: str }`.

### After Phase 5 — branching transitions

`emit_verdict` gets a real decision:

```yaml
emit_verdict:
  transitions:
    - to: done
      when: "output.verdict == 'COMPLETE'"
    - to: check_coverage   # loop back if INCOMPLETE — re-read diff, maybe missed something
      when: "output.verdict == 'INCOMPLETE'"
    - to: done             # NEEDS_DISCUSSION also terminates; humans handle it
```

### After Phase 6 — smoke test passes

Six invocations, walking through a complete run with synthetic data.
Four schema rejection tests confirming the error messages are useful.

Done. The skill is production-grade.

**Total time:** this specific refinement is 30–60 minutes for a
5-state machine. Larger skills (`solve-issue` has ~8 fases) are
1–2 hours. The time scales with the number of states and the
complexity of their outputs, not with the skill's original SKILL.md
size.

## Anti-patterns

**Refining a skeleton you've never used.** You'll guess wrong about
which fields matter, which transitions are real branches vs.
accidental flags, and what the right state granularity is. Use first,
refine second.

**Over-strict output schemas.** If a field is sometimes empty
(`comments: []`), don't mark it required. If a field is sometimes
absent entirely, don't include it in `required`. The runner rejects
valid-looking output whenever the schema says so — err on the side of
too lax, tighten when you see the agent actually emitting wrong data.

**Branching on outputs the schema doesn't require.** If a transition
has `when: "output.status == 'needs_human'"` but the schema doesn't
declare `status` as a required field with an enum, the runner might
evaluate the transition against missing data. Always anchor `when`
expressions against fields the schema guarantees.

**Pruning structural content as "meta".** `## Process`, `## Steps`,
`## Pipeline` — these look like meta from the outside but are usually
the actual sequence of states. Read the content before deleting. The
converter's `# WARNING` flags are hints, not verdicts.

**Expanding one state into many prematurely.** The converter
sometimes produces one state that covers multiple fases (e.g. the
whole "implement" section). Resist the urge to split it into
`plan_implementation`, `write_code`, `write_tests`, `run_tests` unless
there's a real decision boundary between them. More states = more
turns = higher latency. Split only when branching on intermediate
state is genuinely useful.

**Committing a skeleton and the refinement in separate commits.**
The value of the refinement is seeing the before-and-after. Keep the
regenerated skeleton and the refinement in a **single commit** so
`git log --follow` shows one transition, not two.

## Tools reference

Quick reference for the commands used in a refinement session:

```bash
# Regenerate a skeleton from its SKILL.md (destroys hand-edits!)
python3 $HOME/Code/atomic-gates/lib/import_skill.py \
  skills/my-skill/SKILL.md \
  -o skills/my-skill/skill.yaml

# Validate the current skill.yaml against atomic-gates' schema
python3 -c "
import sys, json, yaml
sys.path.insert(0, '$HOME/Code/atomic-gates/lib')
from schema_validate import validate, ValidationError
schema = json.load(open('$HOME/Code/atomic-gates/schemas/skill-machine.schema.json'))
machine = yaml.safe_load(open('skills/my-skill/skill.yaml'))
try:
    validate(machine, schema)
    print('ok')
except ValidationError as e:
    print(f'INVALID: {e}')
"

# Smoke test — create a new run of the skill
CLAUDE_PLUGIN_ROOT=$HOME/Code/atomic-gates \
CLAUDE_PROJECT_DIR=/tmp/smoke \
python3 $HOME/Code/atomic-gates/lib/runner.py <<<'
{"tool_name":"Skill","tool_input":{"skill":"atomic-gates:my-skill","args":"x=1"}}
'
```

## When the skeleton is good enough as-is

Not every skill needs refinement. If the skill:

- Runs once in a while
- Produces output that's read by a human, not parsed by code
- Has no real branching — it's a linear "do this, then this, then this"
- Has a stable SKILL.md that works fine in production

...then the skeleton is the production version. Refinement would add
complexity without buying guarantees you actually need. Leave it alone.

The point of atomic-gates is enforcement **where enforcement matters**.
Skeletons as first drafts are fine. Refinement is a tool you reach for
when "fine" becomes "frustrated."
